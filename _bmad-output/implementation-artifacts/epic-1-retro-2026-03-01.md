## Successes & Strengths

1. **Contracts consolidation set a clean foundation** — once `src/contracts/` was the canonical namespace, every subsequent story built on it without friction; the "where does this go?" question was eliminated for the rest of the epic
2. **Guard tests provide permanent architectural enforcement** — `test_contract_guards.py` now scans source text for structural violations on every test run; zero regressions possible in guarded paths without a test failure
3. **100% delivery, zero regressions** — 5 of 5 stories complete, test count grew from ~138 to 154 with no failures introduced
4. **Frozen value objects already universally applied** — Story 1.4 audit confirmed 17/17 high-frequency value objects already `@dataclass(frozen=True, slots=True)`; the architecture was well-designed from the start
5. **`match/case` dispatch refactor came out clean** — non-obvious Python behavior (imported constants as capture variables vs value patterns) solved correctly via module-alias pattern; resulting dispatch code is genuinely more readable
6. **Team velocity improved over the epic** — Story 1.5 passed in a single review pass; Story 1.3 was the most complex and accounted for most of the review friction

---

## Challenges & Growth Areas

1. **`from __future__ import annotations` compliance — systemic miss in 4/5 stories**
   - Appeared in code reviews for Stories 1.1, 1.3, 1.4, and 1.5
   - Most commonly missed on: test files, files touched indirectly (updated import sites), renamed files
   - Not a developer problem — a checklist problem; the rule exists but certain operations don't trigger the same review attention

2. **Pure file renames carry a compliance blind spot**
   - Story 1.3 renamed `stt/stt.py` → `stt/transcription.py` as a pure rename (no logic changes)
   - The missing `from __future__ import annotations` traveled with the file undetected
   - Caught in Story 1.4's audit — a gap that crossed story boundaries

3. **Story 1.3 required 3 code review rounds**
   - Highest friction story of the epic
   - Issues found: import ordering, double-nested string annotations, lambda-as-named-variable, missing future imports on test files, module-level deferred import violation
   - Not all issues were related to complexity — several were compliance misses that a pre-flight checklist would catch

4. **Pre-existing `pythonpath` gap caught late**
   - `pyproject.toml` missing `pythonpath = ["src"]` meant 2 tests were silently failing before Epic 1
   - Fixed in Story 1.1 but would have been caught earlier with a CI gate

---

## Key Insights & Lessons

1. **The architecture was sound from the start** — Epic 1 primarily formalised and enforced patterns already intuitively in use; the audit work confirmed compliance rather than correcting violations
2. **Review friction concentrates on compliance rules, not logic errors** — this suggests a tooling or pre-flight checklist opportunity rather than a code quality problem
3. **Story complexity ≠ review rounds** — Story 1.3 (moderate complexity rename + boundary fix) had 3× the review cost of Story 1.4 (higher complexity dispatch refactor + audit)
4. **Guard tests are the right investment for architectural invariants** — once written, they self-enforce permanently without human review attention
